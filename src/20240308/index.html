從我的觀點看HTML的演變

HTML超文本標記語言是用於建立網頁的一種標記語言，最早用於出版業中書籍、雜誌和報紙的排版上。而隨著演變用於Web上。

在不同平台上的系統開發，即使是看似相同的UI/UX，所使用的程式語言卻不相同，因此大大增加了維護的成本。因此後來為了解決跨平台的問題，因此出現了Qt。

Qt

Qt是一個跨平台的軟體，用來建立跨平台的GUI應用程式，可運用在Linux、Windows、macOS、iOS、Android甚至在嵌入式系統中。

但隨著技術發展，現在有基於HTML所開發的跨平台應用程式開發Electron。

Electron

使用Node.js作為後端的和Chomium作為前端顯示介面來建立跨平台的應用程式。

但是對於HTML的演變角度上，從關注於排版顯示，演變成逐漸專注於語意上的表示。其目的從以前的顯示給人看，到現在透過具有語意的element提供給搜尋引擎看。

例如說HTML4用於將字體粗體化的element<b>已經不再建議使用，因為它並沒有提供其他語意，現在反而建議是用其他element來表示。

在現有的規範中，逐步地對element產生新的定義，例如以前HTML4中的underline element <u> 已經轉變成 non-textual annotation，可以理解為用於標註些英文拼錯的單字

另外從結構上也從萬用的<div>延伸出<article>、<section>、<header>和<footer>,諸如此類的演變都是為了強化搜尋引擎對於內容的近一步解析與理解。

而豐富整個畫面的CSS也從使用方法上有所不同。最早的時候強調的是語意，到後來發現有如在寫inline style一樣，後來有BEM來指定每塊區域，到現在我認為tailwindcss是最好的解決方式。 其中最重要的核心其實是Cohesion，當使用BEM的時候，當有兩個相同顯示區塊時，有2種寫法，一種是兩個區塊有著一樣的CSS，違反DRY的準則。選項二將兩者區塊抽象化，使用同一個CSS，這解決了DRY準則的問題。 但如果後來那兩個區塊中其中一個要修改，那到時候不僅要新增CSS同時也要改寫HTML，使得維護上變得麻煩。 不管是哪種方法BEM都是low cohesion，因為他們各自的都有著相似的CSS，BEM最大的好處就是low Specificity。

這些element的演變上，即使沿用以前的方式，網頁依然能正常顯示，但就是少了一小部分致力於研究與探索的精神。

Phase 1: "Semantic" CSS (styling decisions, design decision)
  See that .text-center class? Centering text is a design decision, so this code violates "separation of concerns" because we've let styling information bleed into our HTML.

  The quintessential example of this approach has always been CSS Zen Garden; a site designed to show that if you "separate your concerns", you can completely redesign a site just by swapping out the stylesheet.

  ~~"Separation of Concerns"~~ 
  The HTML is independent; it doesn't care how you make it look, it just exposes hooks like .author-bio that the HTML controls.
  Your CSS on the other hand is not independent; it needs to know what classes your HTML has decided to expose, and it needs to target those classes to style the HTML.

  In this model, your HTML is restyleable, but your CSS is not reusable.
  相似的CSS無法重複使用
  
  ~~"Mixing Concerns"~~
  HTML that depends on CSS.

  Naming your classes in a content-agnostic way after the repeating patterns in your UI (like .media-card) treats your CSS as a dependency of your HTML.

  In this model, your CSS is reusable, but your HTML is not restyleable.


  Phase 3: Content-agnostic CSS components

  My goal at this point was to explicitly avoid creating classes that were based on my content, instead trying to name everything in a way that was as reusable as possible.


  Phase 4: Content-agnostic components + utility classes

  Trying to come up with these component names all of the time is exhausting.
  
  When you make modifiers like .actions-list--left, you're creating a whole new component modifier just to assign a single CSS property. It's already got left in the name, so you're not going to fool anyone that it's "semantic" in any way either.
